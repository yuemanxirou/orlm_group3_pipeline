# 数据校对指南

## **Rejected 人工校验任务说明**



你们只需要编辑自己小组的 `group_*.jsonl`。每行是一道题，每道题只分给一个小组校验；本轮不做交叉验证。

实际行数为：



- `group_1.jsonl`: 2434 行

- `group_2.jsonl`: 1622 行

- `group_3.jsonl`: 1623 行

- `group_4.jsonl`: 2434 行

人工校验的核心目标：判断这道 rejected 题是否 feasible；如果 feasible，找出或复制修正一条完整的参考求解轨迹；如果 infeasible，标注 infeasible。



## **只看这些信息**

- 任务含义：

    - 进行数据合成中，我们通过多个大模型的共识选出通过的数据，但仍有部分数据其推理轨迹完成度不高/推理错误/题目质量，可能因为`auto_reject.reject_reasons` 而落选，现在需要各位进行人工修正挽救模型推理轨迹

每行里主要看三块：

- `question.problem`：原始题目。

- `auto_reject.reject_reasons`：自动拒绝原因。

- `model_responses`：三种模型的原始 CoT \+ COPT 代码。

- `review_guidance`：根据自动报错类型生成的只读提示，帮助判断本行重点看什么。

其他字段如 `review_id`、`problem_id`、`assigned_groups`、`review_guidance`、`provenance` 都是追踪用字段，不需要改。



`model_responses` 中每个模型只保留下面三个字段：



```JSON
{
  "solver_name": "deepseek-v4-pro",
  "model": "deepseek-v4-pro",
  "cot_chain": "【求解思路】\n...\n\n【COPT代码】\n```python\n...\n```"
}
```







## **只填写 human\_review**



不要改 `question`、`auto_reject`、`model_responses` 等原始证据字段。只修改 `human_review`：



```JSON
{
  "reviewer_group": "group_1",
  "problem_valid": null,
  "wrong_model_names": [],
  "corrected_reasoning_trace": "",
  "corrected_code": "",
  "corrected_execution_result": null,
  "notes": "",
  "final_accept": null
}
```



## **人工只做四件事**



1. 判断题目是否 feasible，填写 `problem_valid`。

    - `true`：题目清楚、约束自洽，可以求出可行最优解。

    - `false`：题目 infeasible 或题目本身有问题，例如约束冲突、变量不清、无法确定正确模型。

2. 判断哪些模型错了，填写 `wrong_model_names`。

    - 从 `model_responses[].solver_name` 里选，例如 `["deepseek-v4-pro"]`。

    - 不确定可以留空数组 `[]`，但请在 `notes` 说明。

3. 如果题目 feasible，填写正确参考答案。

    - `corrected_reasoning_trace`：最重要，后续会作为训练参考思路。可以基于 `model_responses[].cot_chain` 中的某条模型 CoT 复制后修正，但必须人工检查正确。

    - `corrected_code`：可运行的 COPT Python 代码。

    - `corrected_execution_result`：运行 `corrected_code` 后得到的结果。

4. 给最终结论，填写 `final_accept`。

    - `true`：题目 feasible，并且你已经给出可信的完整 reasoning、代码和运行结果。

    - `false`：题目 infeasible、题目无效，或无法确认正确答案。

## **两种填写模板**



题目 feasible 时：



```JSON
"human_review": {
  "reviewer_group": "group_1",
  "problem_valid": true,
  "wrong_model_names": ["deepseek-v4-pro"],
  "corrected_reasoning_trace": "这里填写人工确认后的正确建模思路、关键约束、求解状态和答案判断。",
  "corrected_code": "import json\\nimport coptpy as cp\\n...\\nprint('RESULT_JSON:' + json.dumps(result, ensure_ascii=False))",
  "corrected_execution_result": {
    "status": "optimal",
    "objective_value": 123.45,
    "solution": {"x1": 1, "x2": 2},
    "answer": "最优方案为..."
  },
  "notes": "说明主要错误，例如 DeepSeek 漏掉能耗约束。",
  "final_accept": true
}
```



题目 infeasible 或不可用时：



```JSON
"human_review": {
  "reviewer_group": "group_1",
  "problem_valid": false,
  "wrong_model_names": [],
  "corrected_reasoning_trace": "",
  "corrected_code": "",
  "corrected_execution_result": null,
  "notes": "题目约束互相矛盾，判定为 infeasible，无法形成可行最优解训练样本。",
  "final_accept": false
}
```



## **最小要求**



- `problem_valid` 和 `final_accept` 必须填写为 `true` 或 `false`。

- 如果 `final_accept=true`，必须填写 `corrected_reasoning_trace`、`corrected_code`、`corrected_execution_result`。

- 如果 `final_accept=false`，至少在 `notes` 里说明原因。

- 保持 JSONL 格式：一行一个完整 JSON object，不要把一条样本拆成多行。

## **回收文件格式**



每个小组回传自己编辑后的原文件即可，文件名保持不变：



- `group_1.jsonl`

- `group_2.jsonl`

- `group_3.jsonl`

- `group_4.jsonl`

不要新建 Excel、CSV 或多行 JSON；不要把多组结果合并到一个文件。每个文件仍然必须是 JSONL：一行一道题，一行就是一个完整 JSON object。



每行只允许修改 `human_review`，其他字段保持原样。回收时每行的 `human_review` 必须满足下面结构：



```JSON
{
  "reviewer_group": "group_1",
  "problem_valid": true,
  "wrong_model_names": ["claude-opus-4-8", "deepseek-v4-pro"],
  "corrected_reasoning_trace": "题目 feasible 时填写完整 reasoning；题目 infeasible 时留空字符串。",
  "corrected_code": "题目 feasible 时填写可运行 COPT Python 代码；题目 infeasible 时留空字符串。",
  "corrected_execution_result": {
    "status": "optimal",
    "objective_value": 650.0,
    "solution": {"take_A": 1, "take_B": 1, "take_C": 1},
    "answer": "自然语言答案"
  },
  "notes": "说明模型错误、题目 infeasible 原因，或其他需要回收方知道的信息。",
  "final_accept": true
}
```



字段要求：



- `reviewer_group`：必须等于文件名对应的小组名，例如 `group_2.jsonl` 中必须是 `"group_2"`。

- `problem_valid`：必须是布尔值 `true` 或 `false`，不能是字符串。

- `wrong_model_names`：数组；从 `model_responses[].solver_name` 中选择错误模型名。若无法判断，填 `[]` 并在 `notes` 说明。

- `corrected_reasoning_trace`：feasible 题必须填写完整求解思路；可以复制某个模型的思路后人工修正。infeasible 题填 `""`。

- `corrected_code`：feasible 题必须填写可运行 COPT Python 代码，且最后打印 `RESULT_JSON:`。infeasible 题填 `""`。

- `corrected_execution_result`：feasible 题必须填写运行 `corrected_code` 得到的 JSON object；infeasible 题填 `null`。

- `notes`：`final_accept=false` 时必须说明原因；feasible 题建议说明哪些模型错、错在哪里。

- `final_accept`：必须是布尔值。feasible 且已补齐 reasoning/code/result 时填 `true`；infeasible、题目无效或无法确认答案时填 `false`。

回收前请做最小自检：



- 文件能被逐行 `json.loads` 解析。

- 行数不变，不删除、不新增、不换行拆分样本。

- 除 `human_review` 外，`review_id`、`problem_id`、`question`、`auto_reject`、`model_responses`、`review_guidance`、`provenance` 都保持原样。

- `final_accept=true` 的行必须同时有非空 `corrected_reasoning_trace`、非空 `corrected_code`、非空 `corrected_execution_result`。

- `final_accept=false` 的行必须有非空 `notes`。

## **推荐人工校对工作流**



1. 先按组分发 `group_*.jsonl`。

    - 每个 reviewer 只编辑自己组的文件。

    - 不改文件名，不拆行，不转 Excel。

2. 用脚本按 `problem_id` 抽取单题到临时文件。

    - 避免直接在 100MB\+ JSONL 里手改。

    - 临时文件只展示 `question`、`auto_reject`、`review_guidance`、`model_responses`、`human_review`。

3. AI 辅助生成候选修正。

    - 把单题 JSON 交给 AI，让 AI 判断题目是否 feasible、哪些模型错了、是否可复制某个模型的 reasoning/code。

    - 如果 feasible，让 AI 生成 corrected reasoning/code/result 的初稿。

    - 如果 infeasible，让 AI 生成 `notes` 初稿。

    - 人必须复核 AI 输出，尤其是模型假设、变量定义、约束、目标函数和运行结果。

4. 本地运行 `corrected_code`。

    - feasible 题必须实际运行代码。

    - 将 `RESULT_JSON:` 后面的 JSON 填入 `corrected_execution_result`。

    - 运行失败就不要填 `final_accept=true`。

5. 用替换脚本写回原 JSONL。

    - 脚本按 `problem_id` 定位原行，只替换 `human_review`。

    - 保留 `question`、`auto_reject`、`model_responses`、`review_guidance`、`provenance` 原样。

6. 批量校验。

    - 检查 JSONL 每行可解析。

    - 检查行数不变。

    - 检查 `reviewer_group` 和文件名一致。

    - 检查 `final_accept=true` 的行三件套齐全：`corrected_reasoning_trace`、`corrected_code`、`corrected_execution_result`。

    - 检查 `final_accept=false` 的行有非空 `notes`。

7. 回收。

    - 每组只回传编辑后的 `group_*.jsonl`。

    - 负责人跑统一 validator，生成错误清单让小组返修。

建议配三个小工具脚本：



- `extract_review_item.py`：输入 `group_1.jsonl --problem-id xxx`，输出单题 pretty JSON，方便人工或 AI 阅读。

- `replace_human_review.py`：输入 `group_1.jsonl --problem-id xxx --human-review fixed.json`，只替换该行的 `human_review`，其他字段不动。

- `validate_review_file.py`：输入 `group_1.jsonl`，输出缺字段、类型错误、JSON 解析失败、`final_accept` 不一致等问题。

AI 辅助时建议使用固定 prompt：



```Plain Text
你是优化建模人工校验助手。只根据下面 JSON 中的 question、auto_reject、model_responses 判断。
任务：
1. 判断题目是否 feasible。
2. 如果 feasible，找出正确模型或修正一条完整 reasoning + COPT code。
3. 如果 infeasible 或题目缺关键信息，标注 final_accept=false，并写清 notes。
4. 不要修改 question、auto_reject、model_responses、review_guidance、provenance。
5. 只输出 human_review JSON。
```



AI 可以辅助初稿，但不能替代运行代码和人工判断。尤其是目标函数不明确、变量单位缺失、可重复/不可重复选择这类问题，必须人工定性。
